# Campus Form 프로젝트 개발 규칙

## 프로젝트 개요
- **프레임워크**: Spring Boot 3.5.9
- **Java 버전**: 17
- **아키텍처**: DDD (Domain-Driven Design) - Layered Architecture
- **데이터베이스**: MySQL 8.0
- **인증**: 세션 + OAuth2

## 아키텍처 구조

### DDD 컨텍스트 구조 (최상위 디렉터리)
프로젝트는 **4개의 Bounded Context(경계 컨텍스트)**로 구성됩니다:

```
identity/        # Identity Context - 사용자 인증 및 프로필 관리
project/         # Project Context - 프로젝트 설정 관리
recruiting/      # Recruiting Context - 프로젝트 생성 이후 모집 과정
notification/    # Notification Context - 알림 관리
```

### 컨텍스트별 책임 및 범위

#### 1. Identity Context (`identity/`)
- **책임**: 사용자 인증 및 프로필 관리
- **주요 기능**: 로그인 (OAuth2, 세션 관리), 사용자 프로필 조회/수정

#### 2. Project Context (`project/`)
- **책임**: 프로젝트 설정 관리 (프로젝트 생성 이후 설정만 포함)
- **주요 기능**: 프로젝트 생성, 시트 연동 및 동기화, 모집 기간 설정, 관리자 관리(OWNER/ADMIN), 프로젝트 상태 관리 및 전환

#### 3. Recruiting Context (`recruiting/`)
- **책임**: 프로젝트 생성 이후의 모집 과정 관리
- **주요 기능**: 지원자 관리(서류/면접 상태), 지원서 처리, 검색/정렬/즐겨찾기, 댓글 관리, 결과 탭(서류/면접 결과), 스마트 시간표

#### 4. Notification Context (`notification/`)
- **책임**: 알림 관리
- **주요 기능**: 인앱 알림 관리, 알림 타입(시트 동기화 결과, 새 지원자, 댓글 생성, 관리자 추가), 알림 이력 관리(read/unread)

### 각 컨텍스트 내부 레이어 구조 (DDD 기반)
각 컨텍스트는 동일한 레이어 구조를 가집니다:

```
{context}/
  ├── domain/        # 도메인 로직 (비즈니스 규칙)
  │   ├── model/     # 엔티티, 값 객체
  │   ├── repository/ # 리포지토리 인터페이스
  │   ├── exception/ # 도메인 예외
  │   └── event/     # 도메인 이벤트
  ├── application/   # 애플리케이션 서비스 (유스케이스)
  │   └── dto/       # 데이터 전송 객체
  ├── infrastructure/ # 인프라스트럭처 (외부 연동)
  │   ├── persistence/ # JPA 구현체
  │   └── external/    # 외부 API 클라이언트
  └── presentation/  # 프레젠테이션 (컨트롤러)
```

### 레이어 간 의존성 규칙
- **presentation** → **application** → **domain** (단방향 의존)
- **infrastructure**는 **domain**의 인터페이스를 구현
- **domain**은 다른 레이어에 의존하지 않음 (순수 비즈니스 로직)

### 컨텍스트 간 통신 규칙
- **컨텍스트 간 직접 의존 금지**: 한 컨텍스트가 다른 컨텍스트의 도메인 모델을 직접 참조하지 않음
- **이벤트 기반 통신 권장**: 컨텍스트 간 통신은 도메인 이벤트를 통해 수행
- **공유 커널**: `global/` 패키지는 모든 컨텍스트에서 공유하는 공통 기능 (예외 처리, 인증, 설정 등)

## 예외 처리 규칙

### 전역 예외 처리 (중앙 집중식)
**모든 예외는 `global/error` 또는 `global/exception` 패키지에서 한 곳에서 처리합니다.**

1. **도메인 예외**: 각 컨텍스트의 `domain/exception/` 폴더에 정의
2. **전역 예외 핸들러**: `global/error/` 또는 `global/exception/`에 `@ControllerAdvice` 사용
3. **예외 처리 원칙**:
   - 컨트롤러나 서비스에서 직접 try-catch 하지 않음
   - 예외는 상위로 전파하고 전역 핸들러에서 처리
   - 일관된 에러 응답 형식 유지 (ErrorResponse DTO 사용)

## 코딩 스타일 및 네이밍

### 패키지 네이밍
- 소문자만 사용
- **최상위 디렉터리는 4개의 Bounded Context로 구성**: `identity`, `project`, `recruiting`, `notification`
- 각 컨텍스트 내부는 레이어별로 패키지 분리 (`domain`, `application`, `infrastructure`, `presentation`)

### 클래스 네이밍
- **Controller**: `*Controller` (예: `ProjectController`)
- **Service**: `*Service` (예: `ProjectService`)
- **Repository**: `*Repository` (인터페이스), `*JpaRepository` (구현체)
- **Entity**: 도메인 모델명 그대로 (예: `Project`)
- **DTO**: `*Request`, `*Response`, `*Dto` (예: `CreateProjectRequest`)
- **Exception**: `*Exception` (예: `ProjectNotFoundException`)

### 메서드 네이밍
- **Controller**: HTTP 메서드에 맞게 (get, post, put, delete, patch)
- **Service**: 비즈니스 동작 표현 (create, update, delete, find)
- **Repository**: Spring Data JPA 규칙 준수 (findBy, existsBy 등)

## DTO 사용 규칙

### Request/Response 분리
- **Request DTO**: `application/dto/` 패키지에 위치
- **Response DTO**: `application/dto/` 패키지에 위치
- Entity를 직접 반환하지 않음 (항상 DTO로 변환)

### DTO 변환
- Entity → DTO 변환은 Service 레이어에서 수행
- MapStruct 또는 직접 매핑 메서드 사용

## 데이터베이스 및 JPA 규칙

### Entity 작성
- `@Entity` 클래스는 각 컨텍스트의 `domain/model/` 패키지에 위치
  - 예: `identity/domain/model/User.java`
  - 예: `project/domain/model/Project.java`
  - 예: `recruiting/domain/model/Application.java`
- JPA 어노테이션만 사용 (`@Id`, `@Column`, `@OneToMany` 등)
- 비즈니스 로직은 Entity 내부에 포함 가능 (도메인 모델)
- **주의**: 다른 컨텍스트의 Entity를 직접 참조하지 않음 (외래키는 ID만 참조)

### Repository
- 인터페이스는 각 컨텍스트의 `domain/repository/` 패키지에 정의
- 구현체는 각 컨텍스트의 `infrastructure/persistence/` 패키지에 위치
- Spring Data JPA 사용 시 인터페이스만으로 충분

### 트랜잭션
- Service 메서드에 `@Transactional` 사용
- 읽기 전용 쿼리는 `@Transactional(readOnly = true)` 명시

## 보안 및 인증

### JWT 토큰
- 토큰 검증은 `global/auth/` 패키지에서 처리
- Security Filter Chain에서 인증 처리

### API 엔드포인트
- 인증이 필요한 API는 `@PreAuthorize` 또는 Security 설정으로 보호
- 인증 정보는 `@AuthenticationPrincipal`로 주입

## 로깅

### 로그 레벨
- **ERROR**: 예외 발생 시
- **WARN**: 경고 상황 (예: 잘못된 요청)
- **INFO**: 중요한 비즈니스 이벤트 (예: 프로젝트 생성)
- **DEBUG**: 개발 시 디버깅 정보

### 로그 형식
- 의미 있는 메시지 작성
- 예외는 stack trace 포함
- 민감한 정보(비밀번호, 토큰)는 로그에 포함하지 않음

## 주석 작성 가이드

### 주석 작성 원칙
**코드의 의도와 맥락을 명확히 전달하는 것을 우선시합니다. 주석은 코드를 읽는 사람이 빠르게 이해할 수 있도록 돕는 도구입니다.**

### 필수 주석 작성 대상

#### 1. Entity 클래스 및 필드
- **클래스 레벨**: Entity의 목적과 책임을 명확히 설명
- **필드 레벨**: 모든 필드에 대해 다음을 포함
  - 필드의 의미와 용도
  - 다른 컨텍스트 참조인 경우 "다른 컨텍스트 참조이므로 ID만 저장" 명시
  - 제약 조건이나 비즈니스 규칙이 있는 경우 설명
  - Enum 타입인 경우 각 값의 의미 설명
- **예시**:
  ```java
  /**
   * 수신자 ID (Identity Context 참조이므로 ID만 저장)
   */
  @Column(name = "receiver_id", nullable = false)
  private Long receiverId;
  ```

#### 2. 메서드
- **public 메서드**: 모든 public 메서드에 JavaDoc 주석 작성
  - 메서드의 목적과 동작 설명
  - 파라미터 설명 (@param)
  - 반환값 설명 (@return)
  - 예외 발생 가능한 경우 (@throws)
- **비즈니스 로직 메서드**: 복잡한 로직이 있는 경우 단계별 설명
- **예외 처리**: 왜 특정 예외를 던지는지 설명

#### 3. 비즈니스 로직
- **복잡한 도메인 규칙**: 비즈니스 규칙이 복잡한 경우 상세 설명
- **상태 전환**: 상태 변경 로직의 전제 조건과 결과 설명
- **권한 검증**: 권한 체크 로직의 이유와 조건 설명

#### 4. 컨텍스트 간 참조
- **식별자 참조**: 다른 컨텍스트의 Entity를 참조할 때 반드시 주석으로 명시
  - 어떤 컨텍스트를 참조하는지
  - 왜 직접 참조하지 않고 ID만 저장하는지
- **예시**:
  ```java
  /**
   * 관리자 ID (Identity Context 참조이므로 ID만 저장)
   */
  @Column(name = "admin_id", nullable = false)
  private Long adminId;
  ```

#### 5. 외부 연동
- **외부 API 호출**: 외부 시스템과의 통신 로직 설명
- **설정값**: 하드코딩된 값이나 매직 넘버의 의미 설명

#### 6. 성능 최적화
- **비효율적으로 보이지만 의도적인 코드**: 성능 최적화를 위한 의도 설명
- **인덱스 사용**: 인덱스가 필요한 이유 설명

#### 7. 예외 처리
- **예외 발생 조건**: 언제 예외가 발생하는지
- **예외 처리 이유**: 왜 특정 예외를 던지는지

### 주석 작성 스타일
- **JavaDoc 형식**: 클래스와 public 메서드는 JavaDoc 주석 사용 (`/** */`)
- **인라인 주석**: 복잡한 로직이나 중요한 비즈니스 규칙은 인라인 주석 사용 (`//`)
- **한국어 사용**: 모든 주석은 한국어로 작성 (비즈니스 로직 설명)
- **간결하고 명확**: 불필요하게 길지 않으면서도 핵심을 전달

### 주석이 불필요한 경우 (예외)
- **자동 생성 코드**: Lombok이 생성하는 getter/setter
- **매우 단순한 코드**: 변수명과 메서드명으로 충분히 설명되는 경우
- **표준 패턴**: Spring의 표준 어노테이션 사용 패턴 (예: `@Transactional`)

## 핵심 도메인 규칙

### 사용자/권한 모델

#### 역할 정의
- **OWNER**: 프로젝트 전체 권한, 마감하기 버튼 노출 및 동작 가능
- **ADMIN**: 지원자 상태 변경 가능, 결과/스마트시간표 조회 및 편집 가능, 단 "단계 마감/종료/전환"은 불가

#### 권한 핵심 규칙
- **마감하기 버튼은 OWNER에게만 노출/동작**
  - 서류 결과: `서류 마감하기` OWNER only
  - 면접 결과: `면접 마감하기` OWNER only

### 프로젝트 상태(State) 및 전환

#### 프로젝트 상태 Enum (`ProjectState`)
```java
public enum ProjectState {
    DOCUMENT_OPEN,      // 프로젝트 생성 완료, 서류 단계
    DOCUMENT_LOCKED,    // 서류 마감(수정 불가), 결과 확정 직전/직후 상태
    DOCUMENT_DONE,      // 서류 종료 및 모집 공고 종료
    INTERVIEW_OPEN,     // 면접 진행/시간표/평가 가능
    INTERVIEW_LOCKED,   // 면접 마감(수정 불가), 결과 확정 직전/직후 상태
    ALL_COMPLETE        // 면접 종료 및 모집 공고 종료
}
```

#### 상태 전환 플로우
1. **서류 마감하기** (OWNER only) → 최초 1회: "서류 결과 완료" 페이지 → `다음 단계: 면접 설정하기` → `INTERVIEW_OPEN` 또는 `면접 없이 모집 종료하기` → `DOCUMENT_DONE`
2. **면접 마감하기** (OWNER only) → 최초 1회: "면접 결과 완료" 페이지 → `종료하기` → `ALL_COMPLETE`
